 1. What actually happened

 ### Initial direction

 We agreed to create two connected SEQ-N1 prototypes:

 - A — Atelier d’échanges: grouping and conservation.
 - B — Console de transformation: predicting and debugging place-value changes.
 - Both connected through the story of Bip’s invention.

 ### First implementation

 The first versions had different stories and mathematics, but shared the same underlying loop:

 ```text
   Read problem → choose one of three answers → receive feedback → continue
 ```

 They were technically complete, but mechanically too similar.

 ### First owner playtest

 The key feedback was:

 │ “The scenarios are okay, but the game mechanic is exactly the same.”

 This exposed a workflow problem: we had validated content coverage and technical quality before validating mechanical differentiation.

 ### Mechanics research and redesign

 We examined transferable mechanics from real games:

 - Sokoban, Wilmot’s Warehouse, Unpacking: spatial organization and constraints.
 - Human Resource Machine, Opus Magnum: program, execute, inspect, revise.
 - Into the Breach: deterministic previews and ordered consequences.

 This produced two genuinely different loops:

 ```text
   A: manipulate → group/open → reorganize → satisfy a spatial constraint
   B: program → execute → inspect trace → reorder/repair
 ```

 Ordered checkpoints were added to B because arithmetic commands often commute; without checkpoints, command ordering would offer false strategic depth.

 ### Second implementation

 - A became a reversible place-value workbench.
 - B became a command sequencer with slots, checkpoints, execution traces, and debugging.
 - Multiple-choice cards were removed from the central interaction.

 ### Second owner playtest

 The workbench felt better, but the reserve and actions were still too textual and visually disconnected.

 Feedback:

 │ “The actions should feel more intuitive.”
 │ “Make it a more coherent surface.”
 │ “The objects should be understandable without relying on text.”

 ### UX and motion pass

 The reserve and controls were rebuilt as one machine-like surface:

 - Visual representations of pieces, sachets, crates, and pallets.
 - Exchange recipes such as ×10 → ×1.
 - Availability indicators.
 - Source/destination pulses.
 - Blocked-action shake.
 - Tactile button depth.
 - Animated validation and scene transitions.

 A restrained motion audit rejected unnecessary particles and object-flight animations because they would interfere with counting.

 ### Asset-generation pass

 We then used image generation for elements where illustration added genuine value:

 - Three Bip poses.
 - Four inventory objects.
 - Four cinematic intro/outro images.

 A hybrid approach proved best:

 - Generated raster assets: characters, objects, narrative scenes.
 - DOM/CSS: quantities, labels, controls, state, and mathematical feedback.

 Two practical image-generation lessons emerged:

 1. The active model did not support transparent output, so assets were generated against chroma magenta, converted to alpha, cropped, and optimized.
 2. The first sachet contained ten unrelated gear shapes. It had to be regenerated using the actual piece asset as a reference so the visual equivalence
    remained mathematically coherent.

 2. Proposed revised workflow

 Phase 1 — Learning brief

 Define:

 - Learning checkpoints.
 - Misconceptions.
 - Required representations.
 - Safety and accessibility constraints.
 - Narrative premise.

 Do not design full scenes yet.

 Phase 2 — Mechanical divergence gate

 Before implementation, write a comparison table:

 ┌─────────────────────┬─────────────────────────────┬──────────────────────────────┐
 │ Dimension           │ Prototype A                 │ Prototype B                  │
 ├─────────────────────┼─────────────────────────────┼──────────────────────────────┤
 │ Primary verb        │ Move, group, open           │ Program, execute, repair     │
 ├─────────────────────┼─────────────────────────────┼──────────────────────────────┤
 │ Puzzle structure    │ Spatial/state-based         │ Temporal/procedural          │
 ├─────────────────────┼─────────────────────────────┼──────────────────────────────┤
 │ Source of challenge │ Capacity and representation │ Route, order and checkpoints │
 ├─────────────────────┼─────────────────────────────┼──────────────────────────────┤
 │ Mistake recovery    │ Undo/reverse exchange       │ Edit/reorder commands        │
 ├─────────────────────┼─────────────────────────────┼──────────────────────────────┤
 │ Visible result      │ Reorganized quantity        │ Numerical execution trace    │
 └─────────────────────┴─────────────────────────────┴──────────────────────────────┘

 ### Gate

 Implementation should not begin until the prototypes differ in:

 1. Player verbs.
 2. State representation.
 3. Source of challenge.
 4. Error recovery.
 5. Moment-to-moment interaction.

 Different content or visuals alone are insufficient.

 Phase 3 — Graybox mechanic

 Build only:

 - One orientation scene.
 - One complication scene.
 - One independent challenge.
 - Minimal styling.
 - No generated artwork.
 - No complete narrative arc.

 The purpose is to answer:

 │ “Is the player doing something interesting and concept-native?”

 Phase 4 — Early owner playtest

 Test both grayboxes before completing the remaining scenes.

 Ask explicitly:

 - Do A and B feel like different games?
 - Is the player manipulating the mathematical idea or selecting an answer?
 - Is the intended action understandable without explanation?
 - Does a mistake produce useful information?
 - Would another scene feel meaningfully different, or merely repeat the same action?

 ### Gate

 Do not add polish until the owner approves the central mechanic.

 Phase 5 — Scene progression

 After mechanic approval, expand into:

 ```text
   Orientation → exploration → constraint → complication → integration → independence
 ```

 Each scene must change either:

 - the constraint,
 - the available tools,
 - the representation,
 - or the reasoning demand.

 Changing only the number is not sufficient.

 Phase 6 — Interaction-surface pass

 Before decorative styling, verify:

 - Actions are visually connected to the objects they affect.
 - Icons and quantities communicate without relying entirely on prose.
 - Available and unavailable actions are distinguishable.
 - Undo/reset are obvious.
 - The surface feels like one coherent system rather than several cards.

 Phase 7 — Motion pass

 Use motion only for:

 - Press feedback.
 - Source/destination explanation.
 - State transitions.
 - Scene transitions.
 - Rare completion moments.

 Reject animation that makes quantities harder to inspect.

 Motion targets:

 ```text
   Press feedback: 100–160ms
   State entrance: 220ms
   Scene entrance: 220ms
   Reduced motion: opacity-only, approximately 120ms
 ```

 Phase 8 — Asset-generation pass

 Only after mechanics and interaction surfaces are approved.

 Recommended division:

 ```text
   Image generation:
   - Character poses
   - Inventory illustrations
   - Intro/outro scenes
   - Decorative narrative artwork

   DOM/CSS:
   - Numbers and exact quantities
   - Mathematical diagrams
   - Interactive controls
   - State indicators
   - Animations requiring precision
 ```

 ### Asset checklist

 - Consistent reference image and palette.
 - Transparent or chroma-keyable background.
 - Generous crop padding.
 - No generated instructional text.
 - Exact object-count verification.
 - Test at actual rendered size.
 - Regenerate anything mathematically ambiguous.

 Phase 9 — Lightweight prototype validation

 Prioritize speed:

 1. Svelte/type validation.
 2. Catalog validation.
 3. One desktop happy path.
 4. One narrow-screen happy path.
 5. Owner playtest.

 Run extended smoke and robustness work only after the mechanic is selected for further development.

 3. Main workflow change

 The most important adjustment is:

 ```text
   Old:
   Brief → complete implementation → technical validation → owner mechanic feedback

   Proposed:
   Brief → mechanic graybox → owner mechanic gate → complete scenes → UX → motion → assets
 ```

 This moves the highest-risk decision—whether the interaction is actually enjoyable and distinct—to the cheapest point in production.

 Questions to settle before integrating this workflow

 1. Should the mechanical divergence gate become mandatory whenever multiple prototype directions are requested?
 2. Should every prototype receive a three-scene graybox before a complete narrative is written?
 3. Should formal smoke testing wait until after owner mechanic approval?
 4. Should image generation always remain a late-stage, optional pass?
 5. Should the workflow require an explicit owner note such as “mechanic approved for polish” before motion and assets begin?

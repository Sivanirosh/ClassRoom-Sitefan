## Decision summary

The source game asks pupils to infer skip-counting patterns and drag missing number tiles onto a sidewalk so a superhero can continue a race. Ten levels increase in difficulty.

The first implementation retained only the ordered-sequence loop and deliberately removed the race. Owner replay identified that as a semantic loss: the fun came from a ground runner being physically blocked before each missing term while a flying opponent keeps moving. The revised graybox therefore preserves that abstract race relationship. Each course and replay first renders the runner at the starting line, then automatically sends it onto the last valid term before the first hole while the rival moves continuously toward the finish. A correct tile updates the term immediately and releases the runner to the last valid term before the next hole; the final repair sends it through the finish before success. Running out of time loses and offers a same-course replay.

Owner replay then identified a second semantic problem: a generic runner track beside separate numbered markers made the blocking relationship indirect and created two competing surfaces. A first unified-course revision still rendered the terms as signal objects, so the metaphor remained nonsensical: the runner appeared to traverse objects rather than ground. The current profile-view course makes every numeric term a connected section of earth. Intact sections share one level surface; each missing section is visibly fallen and rotated below its outlined breach. The runner’s feet sit on the surface and stop at the last intact section, then a correct answer restores the earth flush with the path before the runner crosses it. The aerial rival lane remains directly above and both lanes use the same `--term-count` grid.

The implementation still replaces the source’s superhero, sidewalk, artwork, copy, level count, exact values, timing, branding, and layout. Its runner, flying human rival and handmade glider, three locations, modular terrain, and outcome illustrations are original generated assets with repository-local provenance; the functional race course remains the same unified segmented grid. The first scene gives the `+3` hint. From the second scene onward there is no hint and no preliminary rule-selection phase: the learner infers the step and applies it directly by placing a number.

## Learning contract used for the graybox

- **Core learning:** a regular arithmetic sequence is controlled by one constant additive step that can be inferred from known neighboring terms and applied in either direction.
- **Successful-pupil statement:** « Je peux trouver le même écart entre les nombres, annoncer ce pas, puis reconstruire toutes les sections manquantes du sentier. »
- **Expected cognitive process:** compare known terms → account for how many intervals separate them → infer a constant signed step → generate the next missing term directly → check it against both neighbors → revise after specific feedback.
- **Boundary:** the prototype uses one-operation arithmetic sequences over natural numbers. The timer is a game challenge and replay condition, not evidence that faster responses mean stronger understanding. It does not teach arbitrary visual patterns, alternating rules, multiplication notation, or algebraic formulas.

## Semantic-fit mapping carried into the graybox

| Learning operation | Pupil/game action | System consequence or inspectable evidence | Scene |
|---|---|---|---|
| Apply a given constant step | Select a number tile and place it in the only open ground breach | The completed neighboring differences remain visible | Orientation |
| Infer the governing step | Choose the missing number directly, without a preliminary rule question | The selected term must satisfy the same interval on both sides; an incorrect tile names the failed relation | Complication and independent challenge |
| Generate and check missing terms | Fill the holes in course order from the shared tile bank | Each accepted tile releases the runner to the next hole while the whole ordered sequence stays visible | All scenes |
| Act under a continuing game consequence | Launch the race, inspect the flying rival’s progress, and repair the current hole before it finishes | The rival advances independently of answers; timeout loses and reloads the same course | All scenes |
| Transfer the invariant to reverse direction | Select a negative step and complete a descending sequence | Final feedback writes the repeated subtraction and preserves the completed sequence | Independent challenge |

## Core mechanic hypothesis

> If each missing term physically stops a ground runner on the preceding valid cell while a flying rival continues toward the finish, and pupils infer the constant step through direct tile placement rather than a separate preliminary question, then the repeated pattern operation can gain urgency and visible consequence without adding interaction friction.

- **Primary verbs:** compare, infer, choose, place, release, check, revise.
- **State:** ordered terms, current blocking hole, available tiles, runner position, independently moving rival, time remaining, and written relation feedback.
- **Challenge:** distinguish one interval from a multi-interval gap, preserve the same signed step across the whole sequence, and do so before the rival finishes.
- **Recovery:** an incorrect tile does not advance the runner and the clock continues; the tile stays in the bank while a written explanation and transient outlined `×` mark identify the failed relation. No retry action or answer lock interrupts ordinary correction. Timeout offers replay of the same authored course rather than changing the mathematics.
- **Kill criterion:** reject the adaptation if pupils mainly guess because of the timer, watch the race instead of the sequence, cannot perceive which hole blocks the runner, or treat speed/winning as the learning evidence.

## Three-scene catalog graybox

1. **Orientation — Pas donné.** The `22 s` course automatically starts on `3, 6, 9, ?, 15` and explicitly names `+3`. Placing `12` releases the runner to the finish.
2. **Complication — Pas à trouver.** The `34 s` course automatically starts on `8, ?, 18, 23, ?, 33`. With no hint or rule-selection phase, the learner directly repairs the blocking holes in order with `13` and `28`. A wrong tile consumes race time and explains the failed interval.
3. **Independent challenge — Sens inverse.** The `32 s` course automatically starts on `63, ?, 49, 42, ?, 28`. With no guidance phase, the learner directly releases the runner with `56` and `35`.

Included: start-line-first automatic race start, continuously moving flying opponent, runner stopped on the valid cell immediately before the next missing term, countdown, win/loss/replay, deterministic authored values, direct click/touch/keyboard placement, forgiving pointer drag-and-drop with a real 20 px target halo, automatic non-blocking mathematical error feedback, specific aria-live writing, and no persistence or network activity.

Excluded: score, ten-level progression, sound, random sequences, analytics, classroom submission, and claims of pupil validation.

## Accessibility and interaction

- Number tiles and open ground breaches are native buttons; click/touch selection followed by placement is the keyboard and touch equivalent to drag-and-drop.
- Each control has a complete accessible label including the number or position.
- Rule signs are written as `+` or `−`; success and error are communicated with text and structure, not color alone.
- At 320 px, the shared grid compacts gaps, type, and term-card chrome but keeps all five or six terms in one aligned row without page-level horizontal overflow; the active slot’s real halo may safely overlap disabled neighbors.
- The first course starts when the learner presses **Jouer** on the untimed introduction. Every later course starts automatically when its scene appears; there is no redundant launch control.
- A missing term’s stop point is the center of its preceding valid cell: `index === 0 ? 0 : ((index - 0.5) / termCount) × 100`; finish remains `100%`. This maps scene one’s missing index 3 to the preceding term at `50%`, and scene two’s missing indices 1 and 4 to the preceding terms at approximately `8.33%` then `58.33%`.
- Normal runner travel uses one constant spatial speed with linear easing: `max(650, distance × 18)` ms. Thus `0→50%` takes `900 ms`, `0→8.33%` takes the `650 ms` floor, and `8.33→58.33%` takes `900 ms`. The same computed duration drives both the linear CSS `left` transition and JS completion timing.
- Reduced-motion mode removes runner interpolation and the 180 ms error shake while preserving the same race clock and outcomes. Runner destinations update without travel in at most 80 ms; the planeur remains visually static at the start while time runs, then appears at the finish only if timeout occurs. The static outlined `×` still identifies an error.
- The current slot is a native button whose actual box extends 20 px beyond the visible term card. Drag-enter feedback uses exact 140 ms transform/border/opacity transitions and is not a hover-only touch effect.
- A wrong attempt never disables tiles or slots and exposes no ordinary recovery button. The tile remains available, the active term shows a non-color-only outlined `×` plus a restrained 180 ms shake, and the visible/aria-live error clears automatically after 800 ms or immediately on the next correct placement.
- This graybox has no equivalent untimed route. Before promotion, replay must determine whether a slower-rival or untimed training option can preserve the same reasoning and dignity for pupils who cannot use a speeded route.

## Approved presentation direction

The catalog identity remains **Le pas caché** while the learner-facing mission is **La Course du Sentier**. A young courier crosses a side-profile mountain path made from numbered sections of earth while a second young human courier follows the aligned upper lane in a compact handmade fantasy glider. A missing number is represented literally: that section of ground has fallen below the surface and leaves an outlined breach. Repairing the arithmetic sequence restores the section flush with its neighbors and lets the runner cross. There are no signal objects or object-lighting story. Concise live HTML copy sits over original generated scenery; there is no lore screen, sound, particle reward, score, or unrelated decorative motion.

The three unchanged mathematical scenes are presented as **Étape 1 · Le départ — Répare la première brèche**, **Étape 2 · Le col — Traverse le col**, and **Étape 3 · Le refuge — Redescends vers le refuge**. Learner-facing mathematical vocabulary consistently uses *suite*, *sentier*, *brèche*, *nombre manquant*, and signed *pas*. Success leads with the inspectable completed sequence and its signed step rather than race speed. The final bilan restores the three lines and records `+3`, `+5`, and `−7`; it contains no score or reward.

The visual system is explicitly light (`color-scheme: light`) and includes both the standard light color-scheme metadata and `darkreader-lock`, because a rejected capture showed a browser extension recoloring the exercise into a dark interface. Its owner-supplied core palette is exact: ink `#345B73`, sky `#99E2F2`, foliage `#9BD991`, sunlight `#F2D06B`, and coral `#F29F80`; off-white is used only as a supporting paper neutral. The art direction uses flat matte fills, dark hand-drawn contours, slightly imperfect organic shapes, and playful fantasy-adventure scenery. Generated characters, environments, terrain, source sheets, extraction script, generation metadata, and hashes are recorded under `assets/`; the supplied reference is not bundled and was used only for broad traits.

The active arena is one side-profile landscape rather than stacked abstract tracks. Round one uses the departure meadow, round two a framed mountain pass, and round three the warm refuge descent, while the gameplay geometry remains identical. A dashed upper lane carries the generated flying rival. Directly below, generated modular earth sections share one model-independent surface line and expose live HTML number badges; the runner wrapper’s feet remain within `5 px` of that line. A missing section appears rotated at least `40 px` below its intact neighbor with a generated stones/grass accent and a dashed coral breach above it. Correct placement changes that same section to intact ground before the runner moves. Generated start/end caps and live start/finish signs anchor the ends. The stopwatch remains one locally positioned HTML/CSS component at the top right: its crown and pusher are anchored to the face, and the numeral plus small `s` unit are centered as one group.

The landing/intro page remains visually minimal and contains no instructional cells. The first active round is the intentional tutorial exception: below its answer dock, it explains three things once—observe the repeated interval, drag a number to `?` (or select the number and then the breach), and retry after an error. Keyboard-specific instructions remain screen-reader-only. The cells disappear from rounds two and three.

The visible vocabulary is intentionally small: scene title, stage markers, timer, start/finish signs, sequence values, the unresolved `?`, the first-scene signed step, answer tiles, necessary controls, and actionable feedback. Object names, repair counts, helper sentences, known-section labels, target sentences, and explanatory race callouts are not rendered as visual chrome. The active breach uses a dashed contour and question mark; a repaired section gains a checkmark, so neither state depends only on color.

The number dock is visually attached to the arena and contains only the signed `+3` teaching input when applicable and the answer tiles. Each answer remains visibly a replacement ground fragment, while its numeral sits on a centered white rounded badge matching the intact path labels instead of competing with the soil texture; the live drag follower preserves the same treatment. Selection/target guidance remains in an `aria-live` screen-reader-only relationship rather than occupying a visible panel. Ordinary written error feedback still appears when it is actionable, retains the single smoke feedback hook and 800 ms recovery, and timeout replay remains below the dock. Each active slot has a stable `path-slot-*` id. Keyboard activation of a number (`MouseEvent.detail === 0`) moves focus to that slot, while pointer and touch selection preserve focus location; `aria-controls`, `aria-describedby`, `aria-current="step"`, completed-step checkmarks, and explicit accessible labels expose the same relationships. The 320 px layout keeps the step badge, all answer tiles, and all terms in one row, preserves the real 20 px drop halo, avoids document overflow, and keeps the first actionable number in the initial 800 px viewport.

## Strategic implementation design

The exercise now treats the complete number-pattern race as one deep, EX-0007-specific in-process module in `race-model.ts`. Its public action surface is deliberately small: `createNumberPatternRace()` returns `current()` and `send(intent, nowMs)`. A transition returns one immutable render snapshot, semantic announcement/focus effects, and whether the browser adapter must request another time update. Explicit monotonic time is input data; the module creates no timers, animation frames, DOM nodes, media queries, storage, or network activity.

Behind that interface, the model owns and tests the knowledge that previously crossed many Svelte callbacks: authored-scene validation; deriving the constant signed step from complete terms; ordered-hole validation; available-number and repair projections; runner stop geometry and distance-based duration; first-frame start-line staging; runner arrival; wrong-answer expiry; the authoritative race deadline; timeout/replay; reduced-motion rival behavior; final-travel-before-success; stage progression; completion summaries; and pupil-facing race feedback. A placement at the exact deadline belongs to timeout, while a placement received before it may finish the course. Mathematical repair order, not animation completion, remains the input gate: the existing validated interaction still permits a learner to reason ahead while the runner visually catches up.

`Prototype.svelte` is the thin browser adapter. It owns only Svelte rendering/lifecycle, RAF scheduling, `matchMedia`, semantic-focus-to-selector mapping, native drag payloads and the real 20 px halo, a pointer-tracking visual drag follower, source/target/landing presentation state, aria-live DOM writing, and smoke-contract attributes. It does not calculate answers, steps, destinations, durations, deadlines, feedback expiry, or story progression. The adapter rewrites the live region through an empty state before repeated announcements so the same recoverable error remains perceivable, and clears it when the 800 ms error state expires.

The effective presentation lives in one imported `prototype.css` plus repository-local generated assets. Every selector is rooted under the dedicated `.ex0007-root` class, and every global keyframe carries an `ex0007-` prefix, so loading another prototype cannot inherit this exercise's visual vocabulary. The scenery, three-frame sprite opacity cycles, ground sections, breaches, outcome illustrations, and tactile number dock are presentation-only; the race model has no knowledge of palette or artwork. `assets/extract-assets.py` deterministically removes the chroma backgrounds, decontaminates edges, and normalizes sprite canvases; `assets/provenance.json` records source/generator/output hashes. The asset layer remains EX-0007-local rather than becoming a reusable theme or animation runtime.

The deliberate stopping point is one deep race model, one browser adapter, and one namespaced stylesheet. There are no `Timer`, `Runner`, `FeedbackManager`, scene-controller classes, event bus, injected clock/scheduler ports, generic number-pattern framework, or shallow presentational components. A broader runtime is unjustified until a second real exercise demonstrates the same variation needs.
